# 003. 상태 라이브러리 — 도입 보류 (도입 신호 정의)

- 상태: 채택
- 날짜: 2026-07-07

## 맥락

MVP 플로우(테스트 → 생성 → 편집 → 공유)는 대부분 "한 번 쓰고 한 번 읽는" 흐름이고, App Router에서는 서버 데이터를 Server Component가 직접 읽는다. 상태 라이브러리를 처음부터 깔지, 필요할 때 넣을지 결정이 필요했다.

## 결정

**Zustand · TanStack Query · React Hook Form 모두 지금은 도입하지 않는다.** 대신 도입 신호를 미리 합의해두고, 신호가 오면 팀 합의로 도입한다.

## 근거

- 현재 전역으로 공유할 클라이언트 상태가 없다 — URL(세션/보드 ID)과 로컬 `useState`로 충분
- 서버 캐시가 필요한 화면이 아직 생성 job 폴링 한 곳뿐 — 라이브러리 가치(캐시·무효화·중복 제거)가 발휘될 데이터가 없음
- 6일 MVP에서 안 쓰는 추상화는 학습·리뷰 비용만 늘림. 단, "각자 다른 방식으로 재구현"을 막기 위해 도입 기준 자체는 문서로 못박음

## 결과

도입 신호 ([convention/state.md](../convention/state.md)에 명문화):

- **Zustand** ← 전역 클라이언트 상태(캔버스 편집 등)가 prop drilling/Context 한계에 부딪힐 때 — 캔버스 스토어가 1순위 후보
- **TanStack Query** ← 클라이언트 폴링·캐싱·무효화가 2곳 이상 필요할 때 (job 폴링에 재시도·백오프·탭 전환 처리가 붙는 시점)
- **React Hook Form** ← 검증 필드가 많은 폼이 생길 때 (Zod resolver와 함께)

그전까지 선제 도입도, 개별 재구현 난립도 금지.
