# PRD: 추구미 무드보드 서비스

## 1. 개요

### 1.1 배경

사용자가 자신이 "추구하는 분위기(추구미)"를 이미지·키워드 선택 기반의 간단한 테스트로 표현하면, AI가 이를 시각적 무드보드로 자동 생성해주고 사용자가 직접 편집·완성하여 SNS에 공유할 수 있는 서비스.

### 1.2 용어 정의

| 용어          | 정의                                                             |
| :------------ | :--------------------------------------------------------------- |
| 추구미        | 사용자가 지향하는 스타일/분위기 (미적 취향)                      |
| 추구미 테스트 | 이미지/키워드 선택을 통해 사용자의 취향을 수집하는 질문 플로우   |
| 무드보드      | 선택 결과를 바탕으로 AI가 생성한, 이미지·색·키워드로 구성된 보드 |
| 비회원        | 카카오 로그인을 하지 않고 서비스를 이용하는 사용자               |

## 2. 목표

- 사용자가 로그인 장벽 없이 즉시 무드보드 생성 경험을 체험할 수 있게 한다.
- AI 생성 대기 시간을 지루하지 않게 만들어 이탈을 최소화한다.
- 결과물(무드보드 + 성향 그래프)의 SNS 공유를 유도해 자연 유입을 만든다.

### 성공 지표 (제안, 추후 확정 필요)

- 추구미 테스트 완료율 (테스트 시작 → 생성 요청)
- 무드보드 완성률 (편집 완료 → 결과 도달) · 이미지 저장 전환율
- SNS 공유율
- 비회원 → 회원 전환율 (저장/공유 시점 로그인 유도 기준)

## 3. 대상 사용자 및 접근 권한

| 기능                    | 비회원                   | 회원(카카오/구글 SSO) |
| :---------------------- | :----------------------- | :-------------------- |
| 메인 페이지 열람        | ✅                       | ✅                    |
| 추구미 테스트 진행      | ✅                       | ✅                    |
| 무드보드 생성           | ✅                       | ✅                    |
| 무드보드 편집           | ✅                       | ✅                    |
| 결과 열람               | ✅                       | ✅                    |
| 저장한 보드 재방문 조회 | ⚠️ **그 브라우저에서만** | ✅ 어느 기기에서든    |
| SNS 공유 / 이미지 저장  | ✅                       | ✅                    |

> 비회원도 전체 플로우(테스트→생성→편집→결과·공유·저장)를 끝까지 이용 가능하게 하되, 결과물을 다른 기기에서 다시 보려면 로그인이 필요하다는 점을 결과 페이지에서 안내한다.

**게스트 보드는 "당일"에도 "세션 종료 시"에도 사라지지 않는다.** 완성본은 `guest_session_id` 기준으로 서버에 영구 저장되고, **만료 기간은 없다**(§10.1 · §10.4). 게스트가 보드를 잃는 경우는 하나뿐이다 — `guest_session_id`가 브라우저 `localStorage`에만 있으므로, **저장소를 비우거나 다른 기기·브라우저로 옮기면 자기 보드를 찾을 수 없다.** 서버에서 지워지는 것이 아니라 열쇠를 잃는 것이다.

> `GuestSession.expires_at`(§7)은 스키마에만 있고 **코드 어디에서도 읽지 않는다.** 만료를 도입한다면 사전 안내가 먼저다 ([#123](https://github.com/moodme-teo/mood-me-fe/issues/123)).

## 4. 전체 유저 플로우

```
[앱 실행 · 게스트 우선 진입]   ← 로그인 강제 없음
        │                      (프로필 버튼 → 로그인 화면 / 로그인 시 로그아웃 메뉴)
        ├─ 저장된 보드 있음 ──▶ [홈(History) · 저장 보드 목록] ──┐
        │                                                          │ (무드보드 만들기)
        └─ 저장된 보드 없음 ──▶ [메인 페이지] ─────────────────────┤
                                                                   ▼
                                                            [추구미 테스트]
                                                                   │ (3막 선택 진행)
                                                                   ▼
                                                          [무드보드 생성중]
                                                                   │ (AI 생성 완료)
                                                                   ▼
                                            (이탈 확인 → 홈) ◀── [무드보드 편집]
                                                                   │ (완료)
                                                                   ▼
                                                          [최종 결과물 페이지]
                                                          ├─ SNS 공유 (링크 복사)
                                                          ├─ 이미지 저장 (PNG / JPG)
                                                          ├─ 편집으로 돌아가기
                                                          ├─ 처음부터 다시 만들기
                                                          └─ 홈으로

[홈(History)] ──(저장 보드 카드 탭)──▶ 해당 보드 결과 화면 열람
```

## 5. 화면별 상세 명세

### 5.1 진입 및 로그인 (게스트 우선)

> **정책:** 첫 진입에 로그인을 강제하지 않는다. 앱을 열면 곧바로 메인/홈으로 진입하며(게스트 우선), 로그인은 **사용자가 원할 때**만 시도한다.

- **진입 동작:** 앱 실행 → (온보딩 후) 저장된 무드보드가 있으면 홈(History), 없으면 메인으로 바로 이동. 이 시점의 사용자는 게스트 상태.
- **로그인 진입점:** 메인/홈 상단의 **프로필(아바타) 버튼**을 누르면 로그인 화면으로 이동한다. (첫 진입 강제 화면·모달 없음)
- **로그인 화면 구성:** 카카오 SSO, 구글 SSO, "로그인 없이 둘러보기(게스트)" 버튼
- **동작:**
  - 카카오/구글 로그인 성공 시 → 직전 화면(메인/홈)으로 복귀, 세션에 사용자 식별자 저장
  - 게스트 선택 시 → 게스트 세션을 유지한 채 메인/홈으로 복귀
- **로그인 상태 프로필:** 로그인 상태에서 프로필 버튼을 누르면 **계정 메뉴(로그아웃)** 가 노출된다.
- **예외:** 카카오/구글 인증 실패·취소 시 로그인 화면에 에러 토스트 노출, 재시도 가능

### 5.2 메인 페이지

- **경로:** `/`
- **구성:**
  - 서비스(무드보드) 설명 영역
  - 무드보드 예시 이미지 (캐러셀 또는 그리드, 다른 사용자 결과물 또는 큐레이션 샘플)
  - "이어서 만들기" CTA 버튼
  - "무드보드 만들기" CTA 버튼
- **동작:** CTA 클릭 시 추구미 테스트 시작(담기 단계)으로 이동, 새 세션(테스트 세션) 생성
- **비회원 처리:** 별도 제약 없음

### 5.2-A 홈(History) — 저장된 무드보드 목록

> 저장된 무드보드가 1개 이상이면 진입 홈이 메인 대신 History가 된다. (저장 보드가 없으면 메인 페이지)

- **경로:** `/` (저장 보드 존재 시 기본 진입)
- **구성:**
  - 상단 "History" 타이틀 + 프로필(아바타) 버튼
  - "N개의 무드보드를 모았어요" 카운트
  - 저장된 무드보드 카드 목록 — 각 카드는 보드의 미니 콜라주 썸네일 + 성향 이름
  - "무드보드 만들기" CTA
- **동작:**
  - 카드 탭 → 해당 무드보드의 결과 화면(성향·그래프·공유)으로 이동해 열람
  - CTA → 추구미 테스트 시작(새 세션)
- **회원/게스트 차이:** 회원은 계정에 귀속된 보드가 어느 기기에서든 표시된다. 게스트는 이 브라우저의 `guest_session_id`에 묶인 보드만 표시된다 — 서버에 영구 보관되지만 다른 기기에서는 보이지 않는다(§3 · §5.7). 로그아웃 시 초기 홈(메인)으로 이동.

### 5.3 추구미 테스트

> 상세 스펙(카드 36장·그림자 8개·전환 32개·카피·로깅 스키마·AI 해석 프레임): [docs/work/todo/mood-test-questions.md](../work/todo/mood-test-questions.md) — **7/7 확정, 이 문서가 구현 기준**

- **경로:** `/test/:sessionId`
- **구조:** 단발 객관식이 아닌 **3막 선택 메커니즘** (목표 2~3분) — 행동 데이터(무엇을 언제 버렸는지)가 곧 분석 재료
  1. **담기** — 카드 그리드 36장 중 끌리는 12장 담기
  2. **덜어내기 ×2** — 12→8→5장 강제 덜어내기. 남은 5장 = **확신의 가치**
  3. **그림자** — "요즘 나를 무겁게 하는 것" 칩 8개 중 3개
  4. **전환** — 그림자마다 4지선다(뻔한 반대말 1 + 결이 다른 해석 3). 선택 = **열망의 가치**
  5. **최종 대결** — 확신 5장 + 열망 3장 중 무드보드에 남길 5장 선택
- **상단 헤더:** 홈 버튼 · 진척도 프로그레스바 · 프리뷰 보드
  - **홈 버튼과 뒤로가기는 다른 자리에 있다.** 홈 이동은 헤더 좌측(`홈으로 이동`), 단계 이동(이전·다음·되돌리기)은 하단 `TestFooter` 가 전담한다. 첫 화면에서는 "이전" 버튼이 보이지 않는다.
  - **프리뷰 보드 — 확정.** 지금까지 **살아남은 카드**를 미니 콜라주로 깔고, 그 위에 **담긴 장수 카운트 배지**를 얹는다. 덜어내기에서 남은 카드가 보드에 자리를 잡아가는 연출이다 (상실이 아니라 확정의 경험).
    - 화면마다 의미가 뒤집힌다. 담기에서는 고른 카드가 쌓이고, **덜어내기에서는 고른 카드가 빠진다** — 덜어내기의 선택은 "버릴 카드"이기 때문이다.
    - 카드가 빠질 때 그 자리는 비지 않고 나머지가 재배치된다. 연출의 톤·타이밍은 디자인 소관이며 스펙이 규정하지 않는다.
    - 컨테이너는 `role="group"` + 접근성 이름 `완성되어 가는 추구미 무드보드`. `aria-label` 만 있고 role 이 없는 `div` 는 보조기술이 무시한다.
    - **이 위젯이 전하는 내용은 "몇 장 담겼는가" 하나다.** 카드 이미지는 12px 로 겹쳐 깔린 데다 카운트 배지가 덮으므로 `alt=""` 로 둔다. 카드 이름은 본문 그리드가 말한다. 카운트 배지가 `role="status"` 를 갖고 `aria-label`(`담긴 카드 12장`)로 단위를 붙인다 — 숫자만 있으면 "12" 라고만 읽힌다.
    - **본문 그리드의 카드 이름은 버튼의 접근성 이름으로 노출한다** (`<button aria-label={card.label}>` + 이미지 `alt=""`). 이미지에도 이름을 붙이면 같은 이름이 두 번 읽힌다.
- **본문:**
  - 질문 텍스트 + 질문 설명(부가 설명/의도) + step 표시(step1…)
  - 선택지 UI 5종 중 화면별로 하나 적용. **자유 입력은 없다** — 전 과정이 선택으로만 이뤄진다. 무엇을 골랐는지보다 **무엇을 버렸는지**가 분석 재료이기 때문이다.
    1. **이미지 카드 그리드** — 담기(36→12). 탭으로 토글, 정원(12) 도달 시 나머지가 `disabled`. `CardGrid`
    2. **카드 스택** — 덜어내기(12→8→5). 카드를 아래로 끌어 버리고 좌우로 넘겨 미리 본다. **드래그의 대체 경로로 `이 카드 버리기` 버튼을 함께 준다.** `DiscardStack`
    3. **그림자 칩** — 키워드 칩 8개 중 3개. `ShadowChips`
    4. **전환 4지선다** — 그림자마다 1개(단일 선택). `TransitionChoices`
    5. **최종 캐러셀** — 확신 5 + 열망 3, 총 8장에서 **3장을 탈락시켜 5장을 남긴다.** `FinalGrid` · `FinalCarouselCard`

  > **최종 대결은 고르기가 아니라 걸러내기다.** 8장이 전부 선택된 상태로 시작해 우측 하단으로 끌어 하나씩 탈락시킨다. 5장이 남으면 완성 그리드로 바뀐다. `initialDraftForScreen(final)` 이 pool 전체를 반환하는 이유다.
  >
  > **드래그에는 반드시 대체 경로가 있어야 한다** (§11 접근성). 지금 최종 캐러셀에는 없어서, 모션 최소화를 켠 사용자가 테스트를 끝낼 수 없다 ([#130](https://github.com/moodme-teo/mood-me-fe/issues/130)).

- **선택 규칙(다중 선택):**
  - 화면마다 **정확한 선택 수**를 둔다. 담기 12장 · 덜어내기 1차 4장(→8장 생존) · 덜어내기 2차 3장(→5장 생존) · 그림자 3개 · 전환 그림자마다 1개 · 최종 대결 5장.
  - 조건 충족 전에는 "다음"이 비활성이며, 정원에 도달하면 고르지 않은 선택지가 잠긴다.
  - **안내는 토스트가 아니라 선택 영역 하단의 inline 문구다.** 단계마다 무엇을 왜 고르는지 말한다 (§10.2의 문구 표). 최대 초과처럼 즉각적인 피드백만 짧은 토스트나 흔들림 애니메이션을 쓴다.
- **되돌리기(UNDO):** 현재 화면 안에서 마지막 선택 하나를 취소한다. 화면을 넘기지는 않는다 (`draftHistory`). 화면 간 이동은 "이전" 버튼이 맡는다.
- **상위 단계 수정 시 하위 단계 초기화:** 이전 단계를 바꾸면 이후 단계의 **선택지 구성 자체가 달라지므로** 영향을 받는 이후 단계 데이터만 초기화한다. 조용히 지우지 않고 확정 전에 확인을 받는다.

  > 이전 선택을 바꾸면 이후에 고른 내용이 초기화돼요. 변경할까요?
  > [그대로 둘게요] [변경할게요]

  잃을 것이 없으면(이후 단계를 아직 고르지 않았으면) 묻지 않는다.

  **현재 구현은 말없이 지운다** — `commitScreen` 이 상위가 바뀌면 하위를 비운다. 판정 함수(`willResetDownstream`)는 있고 다이얼로그 배선만 없다 ([#131](https://github.com/moodme-teo/mood-me-fe/issues/131)).

- **여정 로깅:** `{selected, dropped_r1/r2, survivors, shadows, transitions, final, dropped_final, toggles}` — 전 과정이 세션에 저장되어 AI 분석 입력이 된다
- **동작:**
  - 선택 완료 시 "다음" 활성화 → 프리뷰 안착 애니메이션 후 다음 질문으로 이동, 진척도·프리뷰 갱신
  - **마지막 화면(최종 대결)** 의 버튼은 5장을 남기기 전에는 `Create`(비활성)이고, 다 남기면 **`무드보드 생성하기 ✨`** 로 바뀐다. 제출 중에는 `생성 준비 중...`.
    - 현재 구현은 최종 화면에서 항상 `Create` 를 보여주고 `무드보드 생성하기 ✨` 분기에는 닿지 못한다 — 최종 화면이 곧 마지막 화면이기 때문이다 ([#129](https://github.com/moodme-teo/mood-me-fe/issues/129)).
  - 생성하기 → 무드보드 생성중 화면으로 이동, 지금까지의 선택 답변을 생성 요청으로 전송
- **상태 저장:** 뒤로가기로 이전 단계 이동 시 이전 선택값 유지 (재선택 가능)

### 5.4 무드보드 생성중

- **경로:** `/test/:sessionId/generating`
- **구성:**
  - 사용자가 선택한 이미지들이 가상의 무드보드 레이아웃 위에 순차적으로 배치되며 채워지는 인터랙션 애니메이션
  - **프로그레스바** — 진행률(%)로 보여준다
  - 프로그레스바 상단 상태 메시지: 위트있는 문구를 일정 간격으로 로테이션 (§10.3)
- **동작:**
  - 백그라운드에서 실제 AI 생성 작업 진행 상태를 폴링하고, 진행률에 맞춰 프로그레스바 갱신
  - 생성 완료 시 자동으로 무드보드 편집 화면으로 이동
  - **재진입:** jobId를 로컬에 보존해 새로고침·뒤로가기 후에도 기존 job을 다시 폴링한다. 새 생성 요청을 만들지 않는다 ([#115](https://github.com/moodme-teo/mood-me-fe/issues/115))
- **예외 — 이미지와 분석은 따로 실패한다 (§10.3).**
  - **이미지 생성·조립 실패:** 에러 화면 + "다시 시도" (추구미 테스트 결과는 유지한 채 재생성 요청). 편집할 이미지가 없으므로 진입할 수 없다.
  - **분석만 실패:** 편집 화면으로 **그대로 이동한다.** 성향 해석은 결과 페이지에서만 쓰이므로 편집을 막을 이유가 없다. 재시도는 결과 페이지의 그래프 자리에 둔다 (§5.6).
  - **재시도는 실패한 갈래만 다시 돈다.** 성공한 쪽을 다시 호출하면 AI 비용을 한 번 더 태운다.

> **진행률은 클라이언트가 만든다.** 이미지 엔드포인트가 동기 호출이라 서버는 세밀한 진행률을 줄 수 없다 — job은 `processing 10%` → `completed 100%` 두 단계뿐이다. 그래서 클라이언트가 예상 소요 시간(60초) 기준으로 상한(92%)까지 채우고, `completed` 신호를 받으면 100%로 올린다. "로딩은 기다림이 아니라 연출"([ai.md](../convention/ai.md))이라는 원칙에 따른 의도적 설계다.
>
> 그래서 E2E는 진행률의 **절대값을 단언하지 않는다.** 10% 이상에서 채워지기 시작한다는 것과 완료 시 이동한다는 것만 본다 (`mood-test-generating.spec.ts`).

**접근성.** 프로그레스바는 `role="progressbar"` + `aria-valuenow` / `aria-valuemax`를 노출한다 (Radix `Progress`가 자동으로 준다). 상태 메시지는 `role="status"`로 읽힌다.

### 5.5 무드보드 편집

> 상세 스펙(도형 목록·배경 옵션·추천 팔레트·저장 포맷): [docs/prd/mood-edit.md](./mood-edit.md) — **이 문서가 구현 기준**
>
> 편집 화면은 스티커·글자·펜·지우개를 얹는 **캔버스 편집기**에서 **이미지 크롭 에디터**로 교체되었다(#102). 아래는 요약이며, 충돌 시 mood-edit.md를 따른다.

- **경로:** `/moodboard/:moodboardId/edit` (저장본 재편집) · `/test/:sessionId/edit` (생성 직후 편집)
  - 생성 직후 편집은 아직 `moodboardId`가 없어 세션 경로를 이어받는다. 진입 때마다 새 id를 발급하되, §5.7 저장 원칙에 따라 **"완료"** 전까지 서버에 쓰지 않는다.
- **편집 대상은 AI가 만든 이미지 한 장이다.** 사진을 불러오거나 고르는 기능은 없다. 무드미 밖의 기능 목록은 mood-edit.md §1의 표를 따른다.
- **구성:**
  - 상단: 뒤로 · "편집" · 완료
  - 중앙: AI가 생성한 보드 이미지를 현재 도형으로 크롭한 미리보기 (배경색 또는 투명 체크보드)
  - 하단: 모드 탭 4개 — **이미지**(확대·구도 초기화) · **도형**(가로 스크롤 리스트) · **배경**(투명·이미지 블러·흰색·검정·직접 선택) · **색상**(이미지에서 추출한 추천 팔레트)
  - 진입 기본값: 도형 탭 · **크롭 안 함** · **투명 배경** — 세로형 콜라주를 진입하자마자 원형으로 자르면 생성 결과가 망가진 것처럼 보인다. 여백을 흰색으로 칠하는 것도 사용자가 고르지 않은 배경색을 대신 정하는 일이다 (현재 구현은 원형 + 투명, 도형만 바꾸면 된다 — [#117](https://github.com/moodme-teo/mood-me-fe/issues/117))
- **동작:**
  - 도형 선택 즉시 미리보기에 반영되며, 이미지의 위치·확대 상태는 유지된다.
  - **완료**는 편집 결과를 커밋하고 결과물 페이지로 이동만 한다. 파일로 내려받는 일은 결과물 페이지의 "이미지 저장"이 맡는다 — "저장"이 편집 종료·서버 커밋·파일 다운로드 셋을 동시에 가리키면 사용자가 헷갈린다 (현재 구현은 "저장" 시트 안에 셋이 섞여 있다, [#118](https://github.com/moodme-teo/mood-me-fe/issues/118)).
  - 완료 시 크롭 결과를 한 장의 평면 이미지로 export해 서버에 저장한 뒤 결과물 페이지로 이동한다. 크롭 보드는 요소 배열이 아니므로 `elements`는 비운다.
  - **편집 이탈:** 드래프트는 보존한다. 명시적으로 "처음부터 다시 만들기"를 고를 때만 지운다 (현재 구현은 뒤로 버튼만 막고 "폐기"라 안내하는데 실제로는 IndexedDB에 남는다, [#119](https://github.com/moodme-teo/mood-me-fe/issues/119)).
  - **재편집 복원:** 결과물 → 편집 왕복 시 직전 구도가 복원돼야 한다. 지금은 평면 이미지만 저장해 복원할 수 없다 ([#116](https://github.com/moodme-teo/mood-me-fe/issues/116)).
  - **엔딩 마찰 제거:** 완료는 결과(절정) 직전에 로그인으로 가로막지 않는다. 게스트는 곧바로 결과 페이지로 이동하고, 로그인 유도는 결과 페이지의 안내 배너로만 처리한다.
- **비회원 처리:** 저장은 게스트 세션 기준으로 임시 저장, 로그인 시 계정에 귀속
- **레거시:** 결과물 페이지의 뷰어(`BoardPreview`)는 과거에 저장된 `sticker`·`text` 요소를 여전히 렌더한다. 신규 보드는 만들지 않는다.

### 5.6 최종 결과물 페이지

- **경로:** `/moodboard/:moodboardId`
- **구성:**
  - 상단: 완성된 무드보드 이미지 — 편집에서 커밋한 크롭 결과(`exported_image_data_url`) 한 장. 레거시 보드만 뷰어가 `elements` 를 합성한다
  - **무드 성향 이름:** 테스트 여정을 바탕으로 부여되는 명명된 성향 — 성향 이름(한글) + 영문명 + 설명 문구. AI 분석의 **유형명(미학 코어 × 인생 테마)**으로 결정된다(§9). 어느 조합에도 맞지 않으면 기본 성향으로 폴백.
  - **무드 성향 그래프 — 막대 스펙트럼으로 확정.** "무드 성향 5축" 제목 아래, 축마다 좌우 양극 라벨과 채워진 막대, 그리고 비중 %를 표시한다. 축은 다섯 개다: 고요↔활기 · 따뜻↔서늘 · 미니멀↔맥시멀 · 빈티지↔모던 · 현실↔몽환 (`MoodVector`의 `calm_energy` · `warm_cool` · `minimal_maximal` · `vintage_modern` · `real_dreamy`). 값 0~1을 백분율로 반올림해 막대 너비와 라벨에 함께 쓴다.
    - 검토했던 ① 레이더 차트는 채택하지 않았다. ② 막대 스펙트럼 안에 있던 "가장 강한/옅은 결 하이라이트"도 넣지 않았다.
    - 구현: `MoodboardResult.tsx`의 `MoodSpectrum` · `MOOD_AXES`.
    - **분석 실패 시 이 자리가 재시도 지점이다** (§5.4 · §10.3). 성향 이름·그래프 대신 `무드 성향을 읽어내지 못했어요. 무드보드는 그대로예요.` + `분석 다시 시도` 를 놓는다. 무드보드 이미지와 저장·공유는 정상 동작해야 한다 — 잃은 것은 해석뿐이다.
    - 분석이 **아직 도는 중**인 상태와 **실패한** 상태를 구별해야 한다. `mood_profile === null` 만으로는 안 되고 `analysis_status` 가 필요하다 (§10.3).
  - **SNS 공유** 버튼 — **이 결과 페이지 링크를 클립보드에 복사한다.** 공유 시트를 띄우지 않는다. (라벨은 §6 F-23 · 코드와 같은 `SNS 공유` 로 통일한다)
  - **이미지 저장** 버튼 — 누르면 형식을 고른다: `PNG로 저장(투명 유지)` · `JPG로 저장(흰 배경)` (현재 구현은 "이미지 내보내기"가 곧바로 PNG를 내려받는다, [#118](https://github.com/moodme-teo/mood-me-fe/issues/118))
- **동작:**
  - **공유는 링크 복사다.** 공유용 썸네일이나 OG 이미지를 따로 만들지 않는다. 무드보드 이미지와 성향 그래프를 한 장에 합친 공유 카드 PNG도 만들지 않는다 — 보여주고 싶으면 이미지 저장을 쓰고, 링크를 보내면 받는 쪽이 이 페이지에서 본다.
    - `navigator.clipboard`가 막히면 `execCommand` 폴백을 탄다. 어느 쪽이든 토스트는 뜬다.
    - Web Share API·카카오 SDK는 쓰지 않는다 (§10.4).
  - 이미지 저장은 **전부 클라이언트에서 끝난다** (§8). `PNG`는 편집에서 커밋한 크롭 결과를 그대로 내려주므로 재인코딩이 없고 투명값이 유지된다. `JPG`는 그 PNG 를 흰 배경 위에 합성해 재인코딩한다. 과거에 저장된 스티커·글자 요소가 있는 보드는 뷰어가 합성해 렌더한다 (§5.5 레거시, [#124](https://github.com/moodme-teo/mood-me-fe/issues/124)).
  - **재진입 액션:** 편집으로 돌아가기(결과↔편집 왕복), 처음부터 다시 만들기(테스트 재시작), 홈으로.
  - **처음부터 다시 만들기는 기존 보드를 지우지 않는다.** 완성본은 History에 남기고 새 테스트 세션만 시작한다. 되돌릴 수 없는 행동이므로 인앱 확인 다이얼로그를 거친다.
- **비회원 처리:** 페이지 진입 시 안내 배너 노출 (저장 유도). "로그인하면 언제든 다시 볼 수 있어요" 는 **무엇이 위험한지 말하지 않는다.** 잃을 수 있는 것을 먼저 말한다.

  > 지금 만든 무드보드는 이 기기에 임시로 저장돼요. 로그인하면 다른 기기에서도 다시 볼 수 있어요.
  > [로그인하고 보관하기] · 보조: [나중에 할게요]

  "이 기기"는 저장 위치가 아니라 **접근 조건**을 말한다 — §10.4 참조.

### 5.7 세션·상태 보존 (프로토타입 미구현 · 실제 서비스 명세)

> 프로토타입은 인메모리로만 동작하여 새로고침·앱 백그라운드 시 진행 상태가 초기화된다. 이는 플로우 시연용 한계이며, 실제 서비스에서는 아래 정책으로 진행 상태를 보존한다. (프로토타입에서는 코드로 구현하지 않고 명세로만 정의)

- **저장 원칙:** **서버에는 "완성본"만 커밋한다. 진행 중 상태(테스트 답변·편집 중 캔버스)는 모두 클라이언트에 보존하는 드래프트로 취급한다.**
  - ① 로그인/비로그인이 동일한 저장 로직을 공유
  - ② 중간 이탈 시 서버에 무의미한 데이터가 쌓이지 않음
  - ③ 선택마다 서버 왕복이 없어 약전파·오프라인에서도 반응이 즉각적
- **진행 중 테스트:** 각 단계의 선택값(여정 로그)을 클라이언트 store로 관리하고 `localStorage`에 하이드레이션한다. 이탈 후 재진입 시 마지막으로 진행한 단계부터 "이어서 만들기"로 복귀. **테스트 완료 시점에만** 서버에 생성 요청한다.
- **편집 중 보드:** 캔버스 요소(좌표·속성)를 `IndexedDB(or localStorage)`에 주기적으로 autosave하여 새로고침·이탈 후에도 편집 상태를 복원한다. (요소가 많아지고 펜 path가 쌓이면 `localStorage`의 용량·동기 API 한계를 넘으므로 편집 드래프트는 IndexedDB를 사용)
  - **base 이미지 예외:** AI 생성 결과(`base_image_url`)는 서버 에셋이므로 로컬 드래프트에는 두지 않고 **참조(URL/ID)만** 보관한다. 로컬에는 요소 메타데이터 + base 참조만 저장하므로 용량은 작게 유지된다.
  - **서버 저장 시점:** 편집의 **"완료"**(§5.5) 시에만 `Moodboard` row를 서버에 생성한다. base 이미지 에셋만 서버 스토리지에 존재.
- **회원/게스트 차이:** 드래프트가 클라이언트 기반이므로 **회원·게스트 모두 동일 기기·브라우저에서만 이어하기가 가능**하다. 차이는 커밋된 완성본의 귀속에서만 발생한다.
  - 회원: 완성본(무드보드)을 계정에 영구 귀속. History에서 재열람.
  - 게스트: 완성본은 `guest_session_id` 기준으로 서버에 저장되며 **만료되지 않는다**(§3). 로그인 시 계정에 귀속. 결과 페이지 배너로 로그인을 유도하는 이유는 소실 방지가 아니라 **다른 기기에서 열람**하기 위해서다. 로컬 드래프트(및 `guest_session_id`)는 브라우저 저장소를 비우면 소실된다 — 그러면 서버의 완성본에 닿을 열쇠를 잃는다.
- **복원 진입점:** 이어할 로컬 드래프트가 있으면 홈/메인 진입 시 "이어서 만들기" 진입점을 노출한다.

> `POST /moodboards/{id}/export`와 `POST /moodboards/{id}/share`는 **만들지 않는다.** 이미지 저장도 공유도 서버가 할 일이 없다 (§5.6).
>
> - **PNG** — `exported_image_data_url`은 원격 파일 URL이 아니라 **PNG data URL 그 자체**이고, `GET /moodboards/{id}` 응답에 실려 온다. 저장은 앵커 하나(`<a download>`)로 끝난다.
> - **JPG** — 알파를 흰색으로 합성해 재인코딩해야 하지만, 이는 순수 2D 캔버스 연산이다 (`compositeOnWhite()`). data URL 은 캔버스를 tainted 시키지 않으므로 `toDataURL("image/jpeg")`가 그대로 통한다. 원격 URL 이었다면 CORS 헤더 없이는 `SecurityError` 로 죽어 **서버 엔드포인트가 필요했을 것이다** — 저장 형식을 data URL 에서 스토리지로 옮긴다면 이 각주를 다시 검토해야 한다.
> - **공유** — 링크 복사다.

## 9. AI 생성 파이프라인 개요

> 상세: [mood-test-questions.md](../work/todo/mood-test-questions.md)의 AI 해석 프레임·부록(프롬프트 전문)

1. 테스트 세션의 **여정 로그 전체**(선택·라운드별 탈락·그림자·전환·최종 대결·토글)를 취합
2. 여기서 **두 갈래로 갈라져 서로를 기다리지 않는다** (§10.3). 갈림길은 `runGenerationPipeline`이다.

**이미지 갈래** — `assembleBoard`. job의 `status`를 소유하고 편집 진입 여부를 결정한다.

3. `computePersonaResult(journey)` — **결정론적 순수 함수.** 코어/테마 페르소나 비율을 계산한다. **AI를 부르지 않는다.**
4. 비율 블록 + 여정 해시로 레이아웃 1종을 고르고(`pickLayoutStyle`), 프롬프트를 조립한다 (`buildBoardPrompt`)
5. **gpt-image-2 한 호출**(Elice AX 프록시 · 1024×1536 · quality=medium · 실측 40~70초)로 **콜라주 한 장을 통째로 생성**한다 → `base_image_url`
6. `elements`는 **빈 배열**로 둔다. 타일을 코드로 합성하지 않는다 — 콜라주가 이미 이미지 안에 그려져 있다

**분석 갈래** — `runReportAnalysis`. `status`를 건드리지 않는 fire-and-forget이다.

7. 여정을 AI 페이로드로 변환 (id → 라벨·태그 조인, `not_picked`·`was_obvious_antonym` 포함)
8. **GPT-5 한 호출**(`reasoning_effort: "low"`)로 `mood_profile` 산출: 확신·열망 해석 / 무드 벡터 5축 / 키워드 / 스티커 문구
9. `type_name`(미학 코어 × 인생 테마)은 **GPT-5가 아니라 3번의 `computePersonaResult`가 이미 확정한 값**이다. 여기서 합쳐 저장한다 (ADR 004 §개정)

> **이미지 프롬프트는 GPT-5가 만들지 않는다.** 규칙 기반 `buildBoardPrompt(journey)`가 만든다. 이것이 두 갈래가 독립일 수 있는 이유다 — 만약 이미지가 GPT-5의 출력을 재료로 썼다면 분석이 실패했을 때 이미지도 만들 수 없고, §10.3의 "분석만 실패하면 편집으로 보낸다"는 성립하지 않는다.
>
> 이미지 모델은 **`openai/gpt-image-2`**다 (`elice-ai.ts`의 `GPT_IMAGE_MODEL`). Gemini가 아니다. 텍스트는 GPT-5, 이미지는 gpt-image-2 — 둘 다 Elice AX 프록시를 거친다.

## 10. 엣지 케이스 및 에러 처리

세 가지를 구분해서 읽어야 한다.

- **처리됨** — 코드가 이미 그렇게 동작한다
- **갭** — 정책은 정했으나 코드가 아직 따라오지 않았다 (이슈 번호를 단다)
- **해당 없음** — 그 기능 자체가 없어서 발생할 수 없다. 기능이 생기면 이 줄이 갭이 된다

**선택 개수는 헤더 우측 프리뷰 보드의 카운트 배지가 알린다.** 본문 하단의 `N / M 선택됨` 상태 텍스트는 없앴다 — 같은 수를 두 곳에서 읽게 된다. 배지는 `role="status"` 로 값이 바뀔 때마다 읽히고, `aria-label` 로 단위를 붙인다(`담긴 카드 12장`). 아직 코드에 반영되지 않았다 ([#129](https://github.com/moodme-teo/mood-me-fe/issues/129)).

### 10.3 생성

**생성은 독립된 두 갈래다.** 하나로 묶어 다루면 한쪽 실패가 멀쩡한 다른 쪽까지 버리게 된다.

| 갈래                         | 하는 일                                     | job 상태 소유    | 실패하면                              |
| :--------------------------- | :------------------------------------------ | :--------------- | :------------------------------------ |
| **이미지** `assembleBoard`   | gpt-image-2 → `base_image_url` · `elements` | `status` 를 소유 | `failed` → 편집 진입 불가             |
| **분석** `runReportAnalysis` | GPT-5 → `mood_profile` (성향 이름·5축)      | 건드리지 않음    | 조용히 포기 → 결과 페이지 그래프가 빔 |

**편집 진입 여부는 이미지만 결정한다.** 분석 결과는 결과 페이지에서만 쓰이므로, 분석이 실패해도 편집을 막을 이유가 없다. 반대로 이미지가 실패하면 분석이 성공했어도 편집할 대상(`base_image_url`)이 없어 진입할 수 없다 — 편집 대상은 AI 이미지 한 장이다(§5.5).

| 이미지   | 분석     | 이동        | 다시 시도 버튼 위치                                     |
| :------- | :------- | :---------- | :------------------------------------------------------ |
| 성공     | 성공     | 편집 화면   | —                                                       |
| 성공     | **실패** | 편집 화면   | **결과 페이지**의 성향 그래프 자리 (`분석 다시 시도`)   |
| **실패** | 성공     | 생성중 에러 | 생성중 화면 (`이미지 다시 만들기`) — 분석 결과는 재사용 |
| **실패** | **실패** | 생성중 에러 | 생성중 화면 (`다시 시도`) — 둘 다 다시 돈다             |

- **재시도는 실패한 갈래만 다시 돈다.** 이미지 재시도가 성공한 분석을 다시 호출하면 GPT-5 비용을 한 번 더 태운다. 그 반대도 같다.
- **분석 재시도 버튼을 생성중 화면에 두지 않는다.** 그 시점에 사용자는 이미 편집 화면으로 넘어가 있어 버튼을 볼 수 없다.

> **`analysis_status` 컬럼(마이그레이션) — 구현됨(#122).** `mood_profile === null` 만으로는 **"분석 실패"와 "분석이 아직 안 끝남"을 구별할 수 없었다.** 분석은 이미지와 독립적으로 돌아 `completed` 시점 이후에 채워질 수 있다. `moodboard_generation_jobs.analysis_status`(생성 중)와 `moodboards.analysis_status`(저장 후 — 최초 저장 시 job에서 복사, "분석 다시 시도"가 직접 갱신)로 나눠 기록한다.

| 상황                                 | 처리 방안                                                                                                                                                                                                                                                                            | 상태                                                           |
| :----------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------- |
| **AI 분석 실패 + 이미지 생성 성공**  | 위 표대로 편집으로 보내고 결과 페이지 그래프 자리에서 `분석 다시 시도`로 재시도한다. `moodboards.analysis_status`로 실패를 구별한다                                                                                                                                                  | 처리됨                                                         |

| 부적절하거나 빈 이미지 생성          | 검사하지 않는다. 별도 정책 필요                                                                                                                                                                                                                                                      | 갭                                                             |
| 생성 중 새로고침 / 뒤로가기 / 재진입 | jobId를 로컬(sessionId 기준)에 보존해 재진입 시 새 job을 만들지 않고 기존 job을 이어 폴링한다. 서버도 진행 중(`queued`·`processing`) job이 있으면 새로 만들지 않고 그 job을 돌려준다. 답변을 다시 제출하면(테스트 다시하기) 로컬 보존은 무효화되어 새 job이 만들어진다               | 처리됨                                                         |
| 사용자가 생성 중 다른 테스트 시작    | 세션이 다르므로 job도 다르다. 이전 job은 백그라운드에서 완료된다                                                                                                                                                                                                                     | 처리됨                                                         |

**진행률(%)을 프로그레스바로 보여준다.** 서버가 세밀한 숫자를 못 주므로(job은 `processing 10%` → `completed 100%` 두 단계뿐) 클라이언트가 예상 소요 시간 기준으로 상한까지 채운다. 40~70초를 기다리게 하는 화면에서 막대가 움직이지 않으면 멈춘 것처럼 보인다 — 연출이 목적이다.

상태 메시지는 진행률 위에서 일정 간격으로 로테이션한다.

> 당신이 고른 공기를 모으는 중…
> 색과 결을 섞고 있어요…
> 흩어진 조각들을 이어붙이는 중…
> 빛을 조금 더 얹는 중…
> 거의 다 됐어요, 마지막 손질 중…

장시간 지연: `평소보다 조금 더 정성 들여 만들고 있어요. 화면을 닫아도 다시 이어볼 수 있어요.`
— **재진입이 실제로 동작한 뒤에만**([#115](https://github.com/moodme-teo/mood-me-fe/issues/115)) 이 문구를 쓴다. 지금 쓰면 거짓말이다.

재진입: `만들던 무드보드를 다시 불러오고 있어요.`

**실패 문구는 갈래마다 다르다.** 원인이 다르면 사용자가 할 수 있는 일도 다르다.

| 실패                      | 화면                 | 문구                                                                                    | 버튼                         |
| :------------------------ | :------------------- | :-------------------------------------------------------------------------------------- | :--------------------------- |
| 이미지 생성·조립          | 생성중               | `무드보드를 완성하지 못했어요. 고른 내용은 그대로 남아 있으니 다시 만들어볼 수 있어요.` | `다시 만들어보기` · `홈으로` |
| 분석 (이미지는 성공)      | **결과** 그래프 자리 | `무드 성향을 읽어내지 못했어요. 무드보드는 그대로예요.`                                 | `분석 다시 시도`             |
| `queued` 정체 (인계 실패) | 생성중               | `생성을 시작하지 못했어요. 다시 시도해 주세요.`                                         | `다시 시도`                  |

- **`queued` 정체에 "AI가 오래 걸린다" 류의 문구를 쓰지 않는다.** AI 호출은 시작조차 하지 않았다 (§10.3 표). "정성 들여 만들고 있어요" 는 거짓말이 된다.
- 분석 실패 문구는 **무드보드가 무사하다는 사실을 먼저 말한다.** 사용자가 잃은 것은 성향 해석뿐이고, 그 화면에서 보드는 눈앞에 멀쩡히 떠 있다.

> 이미지 실패 문구는 두 안이 오갔다 — "무드보드를 만드는 중 문제가 생겼어요. 고른 내용은 그대로 보관되어 있어요." 와 위 문장. **위 문장으로 확정한다.** 앞의 안은 "문제가 생겼어요" 로 끝나 사용자가 다음에 무엇을 할 수 있는지 말해 주지 않는다.
>
> 구현 완료([#115](https://github.com/moodme-teo/mood-me-fe/issues/115)) — 이전 구현은 "앗, 생성이 잠깐 멈췄어요 / 당신의 선택은 그대로 있어요. 다시 시도하면 이어서 그려드릴게요." 였다. "이어서 그려드릴게요" 는 사실이 아니었다 — 재시도는 이어 그리는 게 아니라 처음부터 다시 만든다. 위 확정 문구로 교체했다.

**공유·저장 문구**

**액션은 두 개뿐이다 — `이미지 저장`(형식 선택) 과 `SNS 공유`(링크 복사).** 문구도 그 둘에서만 나온다.

| 시점                     | 문구                                           |
| :----------------------- | :--------------------------------------------- |
| PNG 저장 성공            | PNG 이미지로 저장했어요.                       |
| JPG 저장 성공            | JPG 이미지로 저장했어요.                       |
| 저장 실패                | 이미지를 저장하지 못했어요. 다시 시도해주세요. |
| 이미지가 아직 준비 안 됨 | 이미지 준비가 아직 끝나지 않았어요.            |
| 공유 = 링크 복사 성공    | 공유 링크를 복사했어요.                        |
| 공유 = 링크 복사 실패    | 링크 복사에 실패했어요.                        |

> **삭제한 문구 — 전부 Web Share API 시절의 잔재다.** 공유를 링크 복사로 확정하면서(§5.6) 아래 상황 자체가 사라졌다.
>
> - `공유 취소` — 네이티브 공유 시트가 없으니 취소할 대상이 없다. `navigator.share` 는 코드에 존재하지 않는다.
> - `공유 실패 → "공유하지 못했어요. 링크 복사를 이용해주세요."` — **순환 문구다.** 공유가 곧 링크 복사인데 링크 복사를 권한다.
> - `앨범 저장 불확실 → "이미지 다운로드를 시작했어요."` — 같은 클릭에 성공 토스트가 이미 뜬다. 게다가 브라우저가 실제로 앨범에 넣었는지는 **감지할 방법이 없어** 이 문구를 띄울 조건을 쓸 수 없다.

**게스트 로그인 유도.** "로그인하면 언제든 다시 볼 수 있어요"는 무엇이 위험한지 말하지 않는다.

> 지금 만든 무드보드는 이 기기에 임시로 저장돼요. 로그인하면 다른 기기에서도 다시 볼 수 있어요.
> [로그인하고 보관하기] [나중에 할게요]

**"이 기기" 표현은 확정됐다.** 게스트 완성본은 `guest_session_id` 기준으로 서버에 저장되지만, 그 id는 브라우저 `localStorage`에만 있다. 기기(또는 브라우저)를 바꾸면 id가 없어 **자기 보드를 찾을 수 없다.** 사용자가 체감하는 것은 저장 위치가 아니라 접근 가능 여부이므로, "이 기기에 임시로 저장돼요"는 사용자 입장에서 정확하다.

이 표현은 게스트 보드가 서버에 없다는 뜻이 **아니다.** 만료 정책([#123](https://github.com/moodme-teo/mood-me-fe/issues/123))을 논의할 때 이 구분을 흐리지 말 것.

## 11. 비기능 요구사항

- **성능:** 생성 대기 화면은 실제 처리 시간과 무관하게 지루하지 않게 인터랙션·메시지로 체감 대기시간을 줄인다.
- **반응형:** 모바일 우선 (SNS 공유 대상이 모바일일 가능성이 높음).
- **접근성:** 편집 도구(도형·배경·확대)는 터치 조작 기준으로 설계. 핀치 줌·드래그 이동·더블탭 초기화는 슬라이더·버튼 대체 경로를 함께 제공한다.
  - **이 규칙은 추구미 테스트의 드래그에도 똑같이 적용된다.** 덜어내기(`DiscardStack`)는 `이 카드 버리기` 버튼을 함께 주지만, 최종 대결(`FinalCarouselCard`)은 드래그뿐이고 그 드래그가 `prefers-reduced-motion` 에서 꺼진다 — **모션 최소화 사용자는 테스트를 끝낼 수 없다** ([#130](https://github.com/moodme-teo/mood-me-fe/issues/130)).
- **피드백 일관성:** 작업 결과(저장·복사·오류 등)는 공통 토스트로, 확인이 필요한 파괴적 동작(편집 이탈 등)은 확인 모달로 일관되게 처리한다.
- **변경 요청 버튼은 잠근다.** `POST`·`PATCH`·`PUT`·`DELETE`를 부르는 버튼은 **예외 없이** 요청이 도는 동안 버튼 안에서 로딩을 돌리고 `disabled`로 잠근다. 페이지 전환이 일어나는 경우에도 전환이 끝날 때까지 잠근 채로 둔다 — 전환 직전의 연타가 요청을 두 번 보낸다. 무드보드 생성은 호출 한 번이 실비용이라 이 규칙이 특히 강하게 적용된다.

  | 버튼                             | 메서드  | 상태                                                                                                                               |
  | :------------------------------- | :------ | :--------------------------------------------------------------------------------------------------------------------------------- |
  | 무드보드 생성하기 (§5.3)         | `POST`  | 처리됨 — `disabled={!canConfirm \|\| isSubmitting}`                                                                                |
  | 완료 (§5.5)                      | `PATCH` | 처리됨 — `disabled={isSaving}` (현재 구현 라벨은 "완성하고 공유하기", [#118](https://github.com/moodme-teo/mood-me-fe/issues/118)) |
  | 다시 만들어보기 (§5.4 생성 실패) | `POST`  | 처리됨 — `disabled={isRetrying}` ([#115](https://github.com/moodme-teo/mood-me-fe/issues/115), 현재 구현 라벨은 "다시 만들어보기") |
  | 다시 시도 (§5.6 로드 실패)       | `GET`   | 갭 — 비용은 없으나 로딩 표시가 없다                                                                                                |

- **개인정보:** 카카오 SSO 시 수집 동의 항목(닉네임, 프로필 이미지 등) 명시.
- **콘텐츠 라이선스:** 예시 이미지·스티커 에셋의 저작권/라이선스 확인 필요.
