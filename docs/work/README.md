# 작업 워크플로우 (docs/work)

무드미 팀은 **이슈 등록 → 배정 → 구현 → PR → 리뷰 → 머지·완료** 흐름으로 작업합니다.
이슈 등록, 구현/PR, 완료 처리를 세 개의 스킬로 자동화했습니다.

```
/work-issue ──▶ GitHub 이슈(명세 본문+라벨+배정+보드+일정) ──/work-work──▶ 구현+커밋+PR ──▶ 리뷰 ──/work-done──▶ 머지·이슈 닫기
 (역할, 작업이름, 일정, 페이지 입력 — 레포 파일·커밋 없음)      (이슈 선택, 미배정이면 집어가며 배정)        (또는 GitHub에서 직접 머지)
```

## 최초 1회

```bash
brew install gh                   # GitHub CLI
gh auth login                      # GitHub.com / HTTPS / 브라우저 인증
gh auth refresh -s project         # 프로젝트 보드(백로그·일정 자동 기록)용 권한 추가
```

> `project` 스코프가 없으면 이슈 생성/배정까지는 되지만 **보드 자동 배치와 일정 기록이 안 됩니다.**
> `error: ... missing required scopes` 가 뜨면 위 refresh 명령을 실행하세요.

## 디렉터리 구조

```
docs/work/
├── README.md              # 이 문서
├── ISSUE_TEMPLATE.md      # 이슈 본문 템플릿 — /work-issue 가 이 구조로 이슈 본문 작성
├── plan/
│   └── todo-list.md       # (폐기 예정) 남은 항목을 이슈로 이관 후 삭제
└── todo/
    └── reference/         # 작업 자료 (수집 이미지 등, 필요 시)
```

## 1. `/work-issue` — 작업 → GitHub 이슈 즉시 등록

```
/work-issue 빌더, 로그인 페이지 구현, 26.07.06~26.07.12, 로그인
/work-issue 빌더, 로그인 페이지 구현, 26.07.06~26.07.12, 로그인 --assignee hyeon-aa
/work-issue 그로워, OG 이미지 개선, 26.07.09~26.07.09, 보드완성 --assignee none
```

- **프로젝트(PRD·코드)를 탐색해 명세(개요/작업 내용/완료 조건)를 이슈 본문으로 작성** — 사용자 확인 후 등록
- 이슈 컨벤션 **시안 A(이모지 브래킷형)**, **페이지 라벨** 부착
- **배정**: 기본은 실행한 본인. `--assignee <로그인명>` 으로 지정, `--assignee none` 으로 미배정(백로그에서 집어가기)
- **무드미 MVP 보드 → 백로그** 추가 + **날짜 필드에 일정 기록** (일정 뷰에 표시) + 역할 필드 기록
- 레포 파일을 만들지 않으므로 **커밋이 필요 없습니다**
- 역할: 💡 프로토타이퍼 · 🚀 빌더 · 🧹 스위퍼 · 📈 그로워 · 🛠 메인테이너 · 🎨 UXUI Leader · 📋 PM · 👬 다같이
- 일정 형식: `26.07.06~26.07.12` / 페이지: 아래 라벨 7개 중 하나

> **여러 개를 한꺼번에 등록**할 때는(계획 회의 등) 스킬 대신 GitHub에서 직접 이슈를 만드세요 —
> `ISSUE_TEMPLATE.md` 구조를 따르면 됩니다. 스킬은 건당 명세 탐색을 수행해 대량 등록에는 느립니다.

## 2. `/work-work` — 이슈 선택 → 구현 & PR

실행하면:

- **나에게 배정된 이슈 + 미배정 이슈** 목록을 불러와 작업할 이슈를 선택 (미배정 이슈를 고르면 본인에게 배정)
- 보드 상태를 `진행중` 으로 이동
- **이슈 본문의 명세**대로 구현 (`dev` 기준 새 브랜치 — prefix는 이슈 제목 `[Feat]` 브래킷에서 역산)
- 커밋 컨벤션(`<prefix> : <메시지>`)에 맞춰 커밋
- `dev` 로 향하는 **PR 생성** — **페이지 라벨** 부착 (`Closes #N` 로 이슈 연결). 리뷰어는 자동 지정하지 않으며, 리뷰 받을 준비가 되면 작성자가 직접 지정합니다 (지정 시 Discord 알림)

이후 **리뷰** 는 사람이 진행하고, 머지·완료 처리는 `/work-done` 으로 합니다 (GitHub에서 직접 머지해도 됩니다).

## 3. `/work-done` — 머지 & 이슈 완료 처리

```
/work-done        # 내 열린 이슈 중 선택
/work-done 42     # 이슈 번호 지정
```

- 이슈에 연결된 PR(`Closes #N`)을 찾아 **`dev` 로 squash-merge** + feature 브랜치 삭제 → 이슈 자동 종료
  - 승인 없이 머지하는 경우라면 한 번 확인을 거칩니다
  - 사람이 GitHub에서 이미 머지했다면 이슈가 닫혔는지만 확인합니다
- **PR이 없는 작업**(조사, 신청 등 레포 결과물 없음)은 이슈만 닫습니다
- 보드 이동은 GitHub Projects 자동화가 처리 (이슈 닫힘 → `작업 완료`)

## 4. 일정관리 · 시각화

별도 인프라 없이 **GitHub Projects(무드미 MVP)** 를 사용합니다.

- **상태**: Board 뷰 — 백로그 → 진행중 → 작업 완료. `/work-issue` 등록 시 `백로그`, `/work-work` 시작 시 `진행중`, `/work-done`(또는 직접 머지)으로 이슈가 닫히면 자동 `작업 완료`.
- **일정**: **`일정` Roadmap 뷰** — 날짜 필드 기준으로 이슈가 타임라인(간트형)에 표시됩니다. `/work-issue` 가 일정을 날짜 필드에 자동 기록하므로 별도 관리가 필요 없습니다.
  - 뷰 설정(⚙️)의 **Date fields** 에 시작/종료 날짜 필드가 지정되어 있어야 합니다.
- **역할**: 보드의 `역할` 필드로 기록 — 역할별 그룹핑/필터 뷰를 만들 수 있습니다.
- **Discord 알림**: 이슈 생성/닫힘/재오픈과 진행중 이동, PR 리뷰어 지정 시 자동으로 채널 알림이 갑니다 (`.github/workflows/discord-*.yml`). 보드에서 카드를 손으로 드래그한 변경은 감지되지 않습니다.

## 브랜치 전략

```
main ───────────────●──────────────▶  release 된 것 (배포)
                     ▲ merge
dev ──●────●────●────●──────────────▶  개발 통합 (feature PR을 squash-merge)
       ▲         │
       │ 브랜치   │ PR → merge
feature ●────────┘                      dev에서 따서 작업
```

- **main**: release 기준. dev가 안정화되면 main으로 **일반 merge** (squash 아님 — dev 커밋 이력 보존).
- **dev**: 개발 통합 브랜치. 모든 feature PR은 여기로 **squash-merge** (`/work-done` 또는 직접).
- **feature**: `dev`에서 따서 작업 → PR 생성 → `dev`로 squash-merge.

> 머지 방식은 브랜치 룰셋으로 강제됩니다: `dev` 는 squash만, `main` 은 일반 merge만 허용.

**브랜치 컨벤션**: `<prefix>/<작업명>` (prefix = 커밋 타입, 작업명 = 영문 kebab-case)

```
feat/login      fix/board-export      refactor/test-flow
```

> `/work-work` 실행 시 이 규칙대로 `dev` 기준 feature 브랜치가 자동 생성됩니다.

## 이슈 / PR 컨벤션 (시안 A)

```
이슈 제목:  [Feat] 로그인 페이지 구현     ← [타입 첫 글자만 대문자] + 제목
PR 제목:    feat : 로그인 페이지 구현     ← 커밋 컨벤션과 동일
라벨:       페이지 단위 — 로그인·메인·테스트·보드생성·보드편집·보드완성·공통
배정:       기본 본인 (--assignee 로 지정/미배정 가능)
리뷰어:     자동 지정 안 함 — 준비되면 작성자가 직접 지정 (지정 시 Discord 알림)
보드:       무드미 MVP · 백로그 (+ 일정·역할 기록)

이슈 본문:
  ## 📋 개요        (배경/근거)
  ## ✅ 작업 내용    (구현 체크리스트)
  ## 🎯 완료 조건    (리뷰/머지 기준)
  ## 🔗 참고         (관련 문서/링크 + 역할·일정 메타 한 줄)
```
